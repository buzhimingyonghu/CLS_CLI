# 当云产品不再需要控制台：从 GUI 到自然语言的交互革命

> 内部分享 · 2026.03

---

## 一、一个场景，看懂变化

凌晨两点，线上告警响了。

**过去（控制台时代）**：你睡眼惺忪打开电脑 → 登录腾讯云控制台 → 进入 CLS → 找到对应日志主题 → 输入检索语句 → 调整时间范围 → 点击搜索 → 看到报错 → 再查上下文 → 再看关联服务……一套操作下来，十几分钟过去了。

**现在（对话时代）**：你拿起手机，在 IM 里打一句：

> "线上 nginx 最近5分钟有没有5xx？帮我查一下原因。"

几秒钟后，AI 回复你：

> "最近5分钟有 23 条 502 错误，全部来自上游服务 payment-svc 超时（平均响应 12s，阈值 3s）。payment-svc 在 01:57 出现了一次 Full GC，耗时 8.7s，建议扩容或调整 JVM 参数。"

**你全程没有打开任何页面，没有写任何命令，没有登录任何控制台。** 你只是用自然语言说了一句话，AI 就帮你完成了日志检索、上下文关联、根因分析。

这就是我们正在经历的变化。

## 二、用户说人话，AI 来干活——背后靠什么？

上面那个场景里，AI 看起来"很聪明"，但它并不是凭空理解了你的日志系统。它的背后其实发生了这些事：

```
用户："线上 nginx 最近5分钟有没有5xx？"
  │
  ▼
AI 理解意图：需要搜索日志，条件是 status>=500，时间范围是最近5分钟
  │
  ▼
AI 自动调用工具（用户不可见）
  │
  ▼
拿到结果 → 分析 → 再调用工具查上下文 → 综合判断
  │
  ▼
用自然语言把结论告诉用户
```

这里的"工具"是什么？就是 **CLI**。

CLI（Command Line Interface）在这个体系里的角色，**不是给人用的命令行工具，而是 AI 的"手和脚"**。就像人类通过鼠标键盘操作控制台一样，AI 通过 CLI 操作云产品。用户完全不需要知道 CLI 的存在，就像你用微信发消息时不需要知道底层走的是 TCP 还是 QUIC。

所以准确地说：

> **CLI 是 AI Agent 时代的产品基础设施。它让 AI 有能力"动手"操作云产品，从而让用户可以用自然语言完成一切。**

## 三、飞书 CLI：昨天刚发生的事

2026 年 3 月 28 日，飞书官方开源了 `lark-cli`。

表面上看，它是一个命令行工具。但飞书的产品意图非常清晰：**这不是给人敲命令的，而是给 AI Agent 装的"飞书操作能力包"**。

安装之后你会发现，你不需要学任何命令。你只需要打开 Claude Code、Cursor、或任何支持工具调用的 AI，用自然语言说话就行：

| 你说的话 | AI 背后做的事（你看不见） | 你得到的结果 |
|---------|------------------------|------------|
| "帮我查下今天的日程" | 调用飞书日历 API | 今天有3个会议，10点产品评审…… |
| "给产品群发一条消息：明天下午3点评审" | 调用飞书消息 API | 消息已发送 |
| "帮我在多维表格里加一行：需求名xxx，优先级P0" | 调用飞书多维表格 API | 已添加 |

用户全程说人话。CLI 是 AI 调用的底层管道，用户无感知。

飞书 CLI 覆盖了 11 个业务域、2500+ API，意味着**AI 可以替你做飞书里几乎所有的事情**。这不是"多了一种操作方式"，这是**交互范式的跃迁**。

## 四、CLS CLI：让 AI 能操作日志服务

我们为腾讯云日志服务（CLS）做了同样的事。

### 4.1 核心理念：用户说人话，CLS CLI 是 AI 的能力底座

CLS CLI 的设计目标不是"让用户学会敲 cls-cli 命令"，而是：

> **让 AI Agent 具备操作 CLS 的完整能力，从而让用户可以用自然语言完成所有日志相关的工作。**

用户在 Claude Code、IM 对话、或任何 AI Agent 中说一句话，AI 理解意图后，在背后自动调用 CLS CLI 完成操作。整个过程对用户来说就像在跟一个懂日志的专家对话。

### 4.2 用户视角：像跟专家聊天一样操作日志

以下是一些真实场景。左边是用户说的自然语言，右边是 AI 背后调用的能力——但用户完全不需要关心右边这一列：

**场景一：日志排查**

| 用户说 | AI 背后调用的能力 |
|-------|----------------|
| "查一下最近15分钟有没有报错" | 日志检索 + 自然语言时间解析 |
| "这条报错的上下文给我看看" | 日志上下文查询 |
| "持续帮我盯着，有新的错误马上告诉我" | 实时 tail 模式 |
| "最近一天的错误量趋势是什么样的" | 日志直方图统计 |
| "把今天的错误日志导出来给我" | 日志下载 |

**场景二：运维部署**

| 用户说 | AI 背后调用的能力 |
|-------|----------------|
| "帮我在这台机器上装个日志采集器" | LogListener 安装脚本生成 |
| "把这三台 web 服务器加到一个机器组里" | 机器组创建 |
| "采集 /var/log/nginx/access.log，用 JSON 格式解析" | 采集配置创建 + 绑定机器组 |
| "看一下采集器的状态正不正常" | LogListener 状态检查 |

**场景三：告警管理**

| 用户说 | AI 背后调用的能力 |
|-------|----------------|
| "当5分钟内错误超过100条时给我发告警" | 告警策略创建 |
| "最近有触发过什么告警吗" | 告警历史查询 |
| "把那个误报的告警策略关掉" | 告警策略删除 |

**在所有这些场景中，用户不需要知道 CLS 控制台的页面在哪、不需要理解检索语法、不需要记任何命令。说人话就行。**

### 4.3 技术架构：两层设计，保证 AI 什么都能做

CLS CLI 采用两层架构，确保 AI 的能力没有天花板：

```
┌─────────────────────────────────────────────────────┐
│                    用户（自然语言）                     │
│         "查一下最近有没有报错"                          │
└─────────────────────┬───────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────┐
│              AI Agent（Claude Code / IM）              │
│         理解意图 → 选择工具 → 组织参数 → 返回结论        │
└─────────────────────┬───────────────────────────────┘
                      │ 调用（用户不可见）
                      ▼
┌─────────────────────────────────────────────────────┐
│                    CLS CLI                            │
│                                                       │
│  Layer 1: 快捷命令（覆盖高频场景，语义清晰）             │
│    日志检索 · 上下文 · tail · 直方图 · 下载              │
│    主题管理 · 告警管理 · 采集器 · 机器组 · 采集配置       │
│                                                       │
│  Layer 2: 通用 API 透传（覆盖 CLS 全部 API）            │
│    任何 Layer 1 没覆盖的能力，都可以通过 API 透传实现     │
└─────────────────────────────────────────────────────┘
```

- **Layer 1（快捷命令）**：7 个域、40+ 条命令，覆盖日志检索、主题管理、告警管理、采集器部署等高频场景。每条命令都设计了合理的默认值和语义化参数，让 AI 能准确调用。
- **Layer 2（通用 API 透传）**：支持所有 CLS API 3.0 接口的直接调用。这意味着即使 Layer 1 没有覆盖的能力（比如数据加工、投递配置），AI 也可以通过 Layer 2 调用原始 API 来完成。

**两层设计的意义在于：AI 永远不会碰到"这个我做不了"的死胡同。**

### 4.4 当前能力矩阵

| 能力域 | 涵盖场景 | 命令数 |
|-------|---------|--------|
| 日志检索 | 搜索、上下文、实时 tail、直方图、下载 | 5 |
| 主题管理 | 增删查、日志集管理 | 5 |
| 告警管理 | 策略增删查、告警历史、通知渠道 | 5 |
| 采集器管理 | 安装、初始化、启停、状态检查、卸载 | 8 |
| 机器组管理 | 增删查、机器状态 | 5 |
| 采集配置 | 增删查、完整流程指南 | 5 |
| 通用 API | 支持所有 CLS API 3.0 接口 | ∞ |

## 五、从产品经理视角看：云产品的形态正在改变

### 5.1 交互范式的三次跃迁

| 时代 | 用户做什么 | 产品是什么 |
|------|----------|----------|
| **控制台时代** | 人打开浏览器，在页面上点击、填写、操作 | 产品 = 一套 Web 页面 |
| **API/SDK 时代** | 开发者写代码，通过 API 调用云服务 | 产品 = 一套 API + 文档 |
| **AI Agent 时代** | 人说一句自然语言，AI 自动完成所有操作 | **产品 = AI 可调用的能力集** |

我们正处在第三次跃迁的起点。

### 5.2 关键洞察：产品的"界面"变了

过去，我们产品经理的核心工作是设计控制台页面：信息架构怎么组织、表单怎么设计、操作流程怎么编排、按钮放在哪里。**产品的"界面"就是 GUI。**

但在 AI Agent 时代，用户不再直接操作界面。用户和产品之间隔了一层 AI。**AI 才是你产品的"用户"**，真正的人类用户只是在和 AI 对话。

这意味着：

**1）产品的"界面"从 GUI 变成了"AI 可调用的能力接口"**

用户不会再关心你控制台的页面好不好看、按钮在哪里。他们只关心：我说一句话，AI 能不能帮我搞定？能搞定，产品就好用；搞不定，产品就不行。

而 AI 能不能搞定，取决于你给 AI 提供的"工具"够不够好——**CLI / MCP Server / API** 就是这些工具。

**2）"好用"的定义变了**

控制台时代的"好用"= 页面美观、交互流畅、学习成本低。

AI Agent 时代的"好用"= **AI 调用你的接口时，语义是否清晰、参数是否合理、默认值是否智能、错误信息是否有用、能力是否完备**。

举个例子：如果你的日志检索 API 只接受 Unix 时间戳，AI 就需要多做一步时间转换，容易出错。但如果你的接口直接支持 "15 minutes ago" 这种自然语言时间，AI 理解用户意图后可以直接传进来，准确率更高。

**这就是为什么 CLS CLI 要做自然语言时间解析——不是为了让人敲命令方便，而是为了让 AI 调用时更准确。**

**3）API 完备性 = 产品竞争力**

当用户习惯了"说一句话就搞定"的体验后，如果你的产品有些功能 AI 做不了（因为 API 不支持），用户就会转向 API 更完备的竞品。

**控制台做得再漂亮，AI 调不了的能力等于不存在。**

### 5.3 产品经理需要怎么适应

| 过去 | 现在 |
|------|------|
| 画原型图、写交互稿 | 设计 AI 可调用的能力接口（CLI 命令语义、API 设计） |
| 思考"用户怎么操作这个页面" | 思考"用户会怎么用自然语言描述这个需求" |
| 优化页面加载速度、交互流畅度 | 优化 AI 调用接口的响应速度、输出结构化程度 |
| 做用户路径分析（页面点击热力图） | 做意图覆盖度分析（用户的自然语言请求有多少能被 AI 处理） |
| 竞品分析看控制台功能对比 | 竞品分析看 API/CLI 能力覆盖度对比 |

## 六、未来的路径

```
现在                          近期                           远期
─────────────────────────────────────────────────────────────────
CLS CLI                →   CLS MCP Server            →   CLS Agent
（AI 通过命令行调用）       （AI 通过协议直接调用）         （AI 自主决策和执行）

用户在 Claude Code        用户在任何 AI 工具中           用户设定目标，
中用自然语言操作 CLS       用自然语言操作 CLS            AI 自动持续运维
```

| 阶段 | 说明 |
|------|------|
| **现在：CLS CLI** | AI Agent 通过执行 CLI 命令操作 CLS，覆盖日志检索、告警、采集等核心场景 |
| **近期：CLS MCP Server** | 将 CLS 能力封装为 MCP 协议（Model Context Protocol），让 Claude Code / Cursor 等工具原生集成，不再需要"执行命令"这个中间步骤 |
| **远期：CLS Agent** | AI 不仅响应单次请求，还能自主持续工作——比如"帮我持续监控线上错误，发现异常自动排查并通知我" |

## 七、总结

**CLI 不是让用户学一套新命令，而是给 AI 装一双能操作云产品的手。**

- 飞书 CLI 证明了：当 AI 有了操作飞书的能力，用户只需要说话就能完成一切
- CLS CLI 做的是同一件事：让 AI 有能力操作日志服务，让用户用自然语言就能完成日志排查、采集部署、告警管理
- 控制台不会消失，但它不再是唯一的产品形态。**能被 AI 调用的能力接口，正在成为云产品新的核心竞争力**

对我们来说，真正要思考的问题不是"CLI 的命令怎么设计"，而是：

> **当用户可以用一句自然语言完成所有操作时，我们的产品准备好了吗？**
